## Finding 1: Invalid-barcode contract is left as a decision instead of a counted interface change
- **Severity:** High
- **Location:** Phase 1, section "Implementation Steps"
- **Flaw:** The plan says to "Decide API behavior" between 400 and 404, but this is an HTTP contract change on an authenticated public route and the plan does not enumerate current backend consumers/tests that assert the route behavior.
- **Failure scenario:** Implementation chooses the preferred 400 path. Mobile or API tests that currently treat lookup misses as "product not found" get a new status family for malformed scanner output. Because the route only returns 404 when the handler returns `None`, the new 400 must be thrown before or inside handler dispatch; if not consistently mapped, invalid input can still fall into provider calls or produce 500 through an uncaught custom exception.
- **Evidence:** Phase 1 leaves the decision open at `phase-01-current-contract-baseline-and-gtin-normalization.md:83-86` and only mitigates with "confirm client handling" at `phase-01-current-contract-baseline-and-gtin-normalization.md:114-115`. The live route maps only `None` to 404 and otherwise builds the response DTO: `src/api/routes/v1/foods.py:44-59`. The query object accepts raw `barcode: str` with no validation: `src/app/queries/food/lookup_barcode_query.py:10-13`. Current route tests patch `get_food_search_event_bus` in 5 places but do not cover invalid barcode behavior: `tests/integration/routes/test_foods_api.py:50`, `tests/integration/routes/test_foods_api.py:75`, `tests/integration/routes/test_foods_api.py:91`, `tests/integration/routes/test_foods_api.py:118`, `tests/integration/routes/test_foods_api.py:134`.
- **Suggested fix:** Make the status decision part of the plan, not implementation discretion. Add exact route-level tests for malformed barcode, bad check digit, and valid-but-not-found, and specify where the exception is translated to `HTTPException(400)`.

## Finding 2: Canonical alias lookup can silently return and cache the wrong barcode form
- **Severity:** Critical
- **Location:** Phase 1, section "Implementation Steps"
- **Flaw:** The plan says to try canonical aliases and "only upsert one canonical barcode", but it does not define the invariant for response barcode vs cache key vs provider barcode. The current schema requires `barcode`, current cache lookup is exact, and current upsert conflicts only on `barcode`.
- **Failure scenario:** User scans UPC-A `123456789012`; alias lookup finds an existing EAN-13 row `0123456789012`. The handler returns the cached row with `barcode="0123456789012"` even though the user scanned `123456789012`, or later upserts the same product under a different alias. Downstream clients see barcode changes between scans, and the global catalog accumulates duplicate rows because the unique key is the raw `barcode` field.
- **Evidence:** Phase 1 introduces aliases at `phase-01-current-contract-baseline-and-gtin-normalization.md:54-64`, says to update cache lookup order at `phase-01-current-contract-baseline-and-gtin-normalization.md:87`, and mitigates duplicate upserts only with a one-write test at `phase-01-current-contract-baseline-and-gtin-normalization.md:112-113`. The live response requires a barcode: `src/api/schemas/response/barcode_product_response.py:13-16`. The repository reads exactly one barcode value: `src/infra/repositories/food_reference_repository_async.py:37-45`. Upsert conflict is only `FoodReferenceModel.barcode`: `src/infra/repositories/food_reference_repository_async.py:102-108`, and the DB model has only a unique nullable `barcode`, no canonical/alias table: `src/infra/database/models/food_reference_model.py:28-36`.
- **Suggested fix:** Specify a single canonical storage form and a separate scanned/raw response policy. Add tests for UPC-A/EAN-13 alias hit, canonical upsert, response barcode value, and no duplicate row under concurrent alias scans.

## Finding 3: FDC provider failure is required to degrade, but the plan omits the catch boundary
- **Severity:** Critical
- **Location:** Phase 2, section "Success Criteria"
- **Flaw:** The plan requires missing key or HTTP failure to fall through to Brave/AI, but implementation steps only add a new service method and cascade step. Current USDA client raises on HTTP errors; the barcode route does not catch handler exceptions.
- **Failure scenario:** Production has `USDA_FDC_API_KEY` missing, expired, rate limited, or FDC returns 5xx. The new cascade step raises from `FoodDataService._get`, aborts the handler, and the barcode endpoint returns 500 instead of continuing to Brave or AI. This creates a cascading outage where adding a fallback makes all misses fail harder.
- **Evidence:** Phase 2 states degradability at `phase-02-usda-fdc-branded-provider.md:107`, but the actual Phase 2 steps only say implement and insert the cascade at `phase-02-usda-fdc-branded-provider.md:73-90`. The live USDA client always calls `resp.raise_for_status()` and returns JSON: `src/infra/adapters/food_data_service.py:22-35`. The live barcode route has no try/except around `event_bus.send`: `src/api/routes/v1/foods.py:51-59`. Current handler only catches FatSecret name-search exceptions and AI-estimate exceptions; direct FatSecret/OFF calls are not wrapped: `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:127-161`, `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:200-229`, `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:337-339`.
- **Suggested fix:** Add an explicit handler-level FDC `try/except` contract: any `FoodDataService` exception records `usda_fdc_error` and falls through. Add a test where `get_branded_food_by_gtin` raises and Brave/AI is still called.

## Finding 7: Source-confidence policy ignores that cache reads overwrite source and lose provenance
- **Severity:** High
- **Location:** Phase 3, section "Success Criteria"
- **Flaw:** The plan treats `cache` as trusted enough to cache/return, but the live handler overwrites every cached hit's original source with `cache`. After FDC and Brave policy changes, clients and metrics cannot distinguish cached FatSecret, cached OpenFoodFacts, cached USDA, or old polluted Brave rows.
- **Failure scenario:** Before rollout, old Brave rows exist in `food_reference` with `source="brave_search"`. After rollout, a scan hits cache first. The handler overwrites the response to `source="cache"` and returns it as trusted, bypassing the new no-Brave-cache policy. Rollout metrics report healthy cache hits while users still receive stale web/LLM nutrition from the global catalog.
- **Evidence:** Phase 3 says cache/fatsecret/openfoodfacts/usda_fdc are "trusted enough to cache" at `phase-03-cascade-confidence-and-brave-demotion.md:45-49` and requires deterministic source values at `phase-03-cascade-confidence-and-brave-demotion.md:94-98`. Live handler overwrites cached source unconditionally: `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:111-117`. The repository persists `source` from cached data: `src/infra/repositories/food_reference_repository_async.py:82-100`, and the model has `source` plus `is_verified`: `src/infra/database/models/food_reference_model.py:53-56`. Current Brave hits are cached today: `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:231-240`.
- **Suggested fix:** Add a cache provenance rule: cached response must expose both retrieval source and original provider source, or at minimum refuse cached rows whose persisted source is now untrusted. Include a migration/backfill or invalidation decision for existing `brave_search` rows.

## Finding 8: Phase 4 forbids high-cardinality metrics but leaves raw barcode logging in the hot path
- **Severity:** Medium
- **Location:** Phase 4, section "Security Considerations"
- **Flaw:** The phase only says not to put raw barcode values in metric labels. Current hit/miss/warning/debug logs include raw barcode values throughout the handler and adapters; adding miss reasons without changing those logs does not satisfy the stated privacy and cardinality intent.
- **Failure scenario:** Rollout adds `invalid_barcode`/provider miss counters but production logs still emit every scanned barcode on hits, misses, provider failures, cache failures, and AI failures. Any log aggregation keyed or searched by message contents effectively becomes high-cardinality barcode telemetry, and malformed barcode input can be retained in logs.
- **Evidence:** Phase 4 mitigation only addresses metric labels at `phase-04-missing-product-capture-and-rollout.md:116-117`, while security says barcode values may be user input at `phase-04-missing-product-capture-and-rollout.md:123-126`. It also logs raw barcode in debug and warning paths: `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:114`, `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:136`, `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:156`, `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:371`, `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:385`. Brave adapter logs raw barcode too: `src/infra/adapters/brave_search_nutrition_service.py:51-53`, `src/infra/adapters/brave_search_nutrition_service.py:88-96`.
- **Suggested fix:** Add a Phase 4 task to scrub or hash barcode values in logs, preserving only safe length/checksum/prefix metadata if needed. Tests should assert miss reason labels do not include raw barcode and log messages do not include full barcode.
