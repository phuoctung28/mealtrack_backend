## Finding 1: Plan leaves raw barcode logging in the production path
- **Severity:** High
- **Location:** Phase 1, section "Security Considerations"; Phase 4, section "Requirements"
- **Flaw:** The plan only says not to log raw malformed input/provider payloads, but the current barcode cascade logs raw barcode values on hits, misses, cache failures, Brave extraction, and AI estimation. Phase 4 adds more miss telemetry without requiring barcode redaction/hashing across existing logs.
- **Failure scenario:** An attacker or curious operator can submit unique barcode-like tokens and later recover a user's scan attempts from app logs. For real scans, logs become a behavioral history of packaged products a user scanned. Phase 4 would make this worse by standardizing miss logging while preserving raw identifiers.
- **Evidence:** Plan says "Do not log raw malformed input beyond safe length-limited metadata" in `phase-01-current-contract-baseline-and-gtin-normalization.md:119-121` and "no raw barcode provider payloads or label images in logs" in `phase-04-missing-product-capture-and-rollout.md:32-36`, but current code logs `query.barcode` in hit logs at `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:97-106`, miss logs at `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:264-273`, cache warnings at `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:367-385`, Brave logs at `src/infra/adapters/brave_search_nutrition_service.py:51-53` and `src/infra/adapters/brave_search_nutrition_service.py:151-156`, and AI prompt/debug context at `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:297-310`.
- **Suggested fix:** Add a required Phase 1 task to replace raw barcode logging with fixed-length hash or suffix-only redaction in handler and Brave adapter before adding Phase 4 telemetry.

## Finding 2: "Verified" cache policy is fake because barcode upsert ignores `is_verified`
- **Severity:** Critical
- **Location:** Phase 2, section "Implementation Steps"; Phase 3, section "Architecture"
- **Flaw:** The plan treats `food_reference` as a verified cache/catalog and asks FDC exact hits to set `is_verified=true`, but the barcode upsert path does not persist `is_verified` at all. Source labels are being used as trust labels while the actual schema flag remains false/default.
- **Failure scenario:** FDC exact hits are cached as `source=usda_fdc` but not marked verified in the DB. Later code or moderation logic that depends on `food_reference.is_verified` sees trusted FDC rows as unverified, while untrusted barcode rows can still sit in the same global catalog. This breaks the plan's trust boundary and makes source strings the only guard.
- **Evidence:** Phase 2 says map `is_verified: true for exact GTIN match` and cache exact FDC hits in `phase-02-usda-fdc-branded-provider.md:78-90`; Phase 3 treats `cache/fatsecret/openfoodfacts/usda_fdc` as "trusted enough to cache" in `phase-03-cascade-confidence-and-brave-demotion.md:43-49`. The DB model has `is_verified` at `src/infra/database/models/food_reference_model.py:53-56`, but `AsyncFoodReferenceRepository.upsert()` only writes barcode/name/macros/source/fdc metadata and omits `is_verified` at `src/infra/repositories/food_reference_repository_async.py:80-110`.
- **Suggested fix:** Make Phase 2 explicitly update barcode upsert to persist `is_verified`, add tests for FDC true and Brave false, and define whether FatSecret/OpenFoodFacts barcode hits are verified or merely externally sourced.

## Finding 3: Name-based Brave verification can poison the global barcode cache
- **Severity:** High
- **Location:** Phase 3, section "Requirements"
- **Flaw:** The plan allows a Brave-discovered product identity to trigger structured provider verification by name, but does not require exact barcode/GTIN agreement before caching under the scanned barcode.
- **Failure scenario:** A malicious or SEO-poisoned web result causes Brave+AI to extract a plausible product name for barcode `X`. FatSecret name search returns a generic first matching food with nutrition. The handler overwrites that result's `barcode` with `X` and caches it globally, so future scans of `X` return attacker-influenced nutrition from `food_reference`.
- **Evidence:** Phase 3 requires "Brave+AI product identity can trigger structured provider verification by name" in `phase-03-cascade-confidence-and-brave-demotion.md:34-38` and says Brave candidates may trigger FatSecret/FDC/OFF name search in `phase-03-cascade-confidence-and-brave-demotion.md:63-79`. Current handler already does this unsafe pattern: it searches FatSecret by `brave_name`, takes the first result with nutrition, sets `fs_item["barcode"] = query.barcode`, overwrites the display name with `brave_name`, and caches it at `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:194-223`.
- **Suggested fix:** For any Brave-derived identity, require exact barcode/GTIN verification before `food_reference` cache writes. Name-only matches should be returned as `is_estimate=true` and uncached.

## Finding 4: Canonical alias plan can leave stale duplicate catalog rows
- **Severity:** High
- **Location:** Phase 1, section "Implementation Steps"
- **Flaw:** The plan says to try canonical aliases and "only upsert one canonical barcode", but it has no migration/backfill/merge step for existing rows already stored under alternate UPC/EAN forms. The schema only enforces uniqueness per raw barcode string, not per normalized GTIN identity.
- **Failure scenario:** Existing cache has row A under UPC-A and row B under zero-prefixed EAN-13 for the same product. After Phase 1, lookup order determines which stale row wins; upserting only the canonical key does not delete or merge the alias row. An attacker can exploit this by causing one alias to cache lower-quality data and rely on lookup order to serve it.
- **Evidence:** Phase 1 requires alias generation and cache lookups in `phase-01-current-contract-baseline-and-gtin-normalization.md:34-38` and says "only upsert one canonical barcode" in `phase-01-current-contract-baseline-and-gtin-normalization.md:110-113`. Current cache lookup is exact string equality at `src/infra/repositories/food_reference_repository_async.py:37-45`, upsert conflict is only on `FoodReferenceModel.barcode` at `src/infra/repositories/food_reference_repository_async.py:102-109`, and the model only has a unique raw `barcode` column at `src/infra/database/models/food_reference_model.py:28-30`.
- **Suggested fix:** Add a Phase 1 data strategy: canonical barcode column/index or alias table, deterministic merge precedence, and tests for pre-existing duplicate UPC/EAN rows.

## Finding 5: Invalid-barcode error handling is underspecified and can become a 500
- **Severity:** Medium
- **Location:** Phase 1, section "Implementation Steps"
- **Flaw:** The plan permits wiring normalization into "the API route or handler" but only describes the route raising `HTTPException(400)`. It does not require a route/domain exception mapping if the handler raises `InvalidBarcodeError`.
- **Failure scenario:** Implementer puts the pure domain normalizer in the handler, raises `InvalidBarcodeError`, and the existing route does not catch it. Malformed input then returns a 500 instead of the promised client-safe 400/404, leaking an internal error path and causing noisy production alerts.
- **Evidence:** Phase 1 recommends a domain helper and says to wire it into "barcode route or handler" in `phase-01-current-contract-baseline-and-gtin-normalization.md:66-87`. The current route only maps `result is None` to 404 and otherwise returns the response; it has no exception mapping for domain validation errors at `src/api/routes/v1/foods.py:44-59`. Existing route tests assert generic handler exceptions become 500 at `tests/unit/api/test_small_v1_routers.py:225-233`.
- **Suggested fix:** Require one explicit boundary: route-level normalization before event bus send, or a route catch that maps `InvalidBarcodeError` to the approved status. Add a route test for malformed barcode.

## Finding 6: USDA FDC mapping cannot satisfy the per-100g barcode contract as written
- **Severity:** Medium
- **Location:** Phase 2, section "Implementation Steps"
- **Flaw:** The plan says to map FDC rows through the existing mapping layer to barcode response fields, but the current mapper produces search/detail DTO shapes (`nutrients.protein`, `macros.protein`) rather than the barcode response's `protein_100g`, `carbs_100g`, `fat_100g` fields. It also preserves USDA reported calories instead of deriving calories from macros.
- **Failure scenario:** Implementer reuses `map_search_item()` for a barcode FDC hit. The handler then returns a dict missing required barcode macro keys, `_has_nutrition()` rejects it, or `BarcodeProductResponse` emits null macros while the plan claims FDC exact hit succeeded. If calories are carried forward from USDA nutrient 1008, the backend-source-of-truth calorie invariant is also violated.
- **Evidence:** Phase 2 requires mapping FDC nutrients to "existing per-100g fields" and deriving calories from macros in `phase-02-usda-fdc-branded-provider.md:35-40`, then lists fields including macros and `source`/`is_verified` at `phase-02-usda-fdc-branded-provider.md:78-90`. Current `BarcodeProductResponse` expects `protein_100g`, `carbs_100g`, `fat_100g` at `src/api/schemas/response/barcode_product_response.py:13-32`; `_has_nutrition()` checks the same keys at `src/app/handlers/query_handlers/lookup_barcode_query_handler.py:276-286`. Existing USDA mapping returns nested `nutrients` and `calories` from FDC nutrient 1008 at `src/domain/services/food_mapping_service.py:82-107` and `src/domain/services/food_mapping_service.py:141-163`.
- **Suggested fix:** Add a dedicated FDC-to-barcode mapper contract that outputs `*_100g`, `fiber_100g`, `sugar_100g`, barcode, source, and verification fields, with calories omitted or derived only from mapped macros.

## Finding 7: Event bus wiring change omits the singleton dependency test from plan scope
- **Severity:** Medium
- **Location:** Phase 2, section "Related Code Files"
- **Flaw:** Phase 2 changes `get_food_search_event_bus()` constructor wiring, but does not list the singleton wiring test that monkeypatches dependency getters and handler constructors. This is exactly the kind of fact-check gap that lets a plan pass structurally but break a production singleton initialization path.
- **Failure scenario:** Implementer injects `food_data_service` and `food_mapping_service` into `LookupBarcodeQueryHandler` but forgets to keep the singleton test in sync. The event bus still initializes in tests through patched objects, but if the constructor signature or local imports drift, production can instantiate heavy services or fail at startup when the singleton is first requested.
- **Evidence:** Phase 2 lists event bus and handler files but not the singleton dependency test in `phase-02-usda-fdc-branded-provider.md:56-69`. The current singleton creates `food_data_service` and `food_mapping_service`, but does not pass them to `LookupBarcodeQueryHandler` at `src/api/dependencies/event_bus.py:210-256`. The test patches these dependency getters and asserts exactly three handlers are registered at `tests/unit/api/test_event_bus_dependency_singletons.py:21-54`.
- **Suggested fix:** Add `tests/unit/api/test_event_bus_dependency_singletons.py` to Phase 2 related files and require an assertion that the barcode handler receives the USDA/mapping dependencies without instantiating extra services.
